# Path Traversal

**Date:** March 13, 2026  
**Vulnerability:** Path Traversal (CWE-35)  
**OWASP Category:** [A01:2025 – Broken Access Control](../reference/owasp-top-10.md)

---

## What Was Demonstrated

In this session, we explored **Path Traversal** — a vulnerability where user-supplied input is used to construct file paths without proper validation, allowing attackers to read or write files outside the intended directory.

We built a Spring Boot app with a `/view` endpoint that takes a `filename` parameter. Using the vulnerable version, an attacker could traverse out of the intended directory and read sensitive files like `/flag.txt` or `/etc/passwd`. We then fixed the code using `Path.normalize()` and validated the resolved path stays within the base directory.

We also demonstrated how **running a container as root** amplifies the impact — a path traversal bug running as root can read any file on the system, while a non-root user is restricted by file permissions.

## Vulnerability Details

### Vulnerable Code

The `/view` endpoint concatenates user input directly into the file path using `File()`. Even though there's a prefix check with `startsWith`, using `java.io.File` does **not** normalize the path — `../` sequences can bypass the check depending on how the path resolves.

```java
@GetMapping("/view")
public ResponseEntity<Resource> readFile(@RequestParam String filename) {
    File file = new File(BASE_PATH + filename);
    if (!file.getAbsolutePath().startsWith(new File(BASE_PATH).getAbsolutePath())) {
        throw new SecurityException("Access Denied!");
    }
    if (!file.exists()) {
        return ResponseEntity.status(404).body(null);
    }
    return ResponseEntity.ok()
            .contentType(MediaType.TEXT_PLAIN)
            .body(new FileSystemResource(file));
}
```
### Intended Function

```bash
# Read the public file that is the intented function of the application
curl "http://localhost:8080/view?filename=public.txt
```
### Exploitation

```bash
# Read the flag file planted outside the app directory
curl "http://localhost:8080/view?filename=../../../flag.txt"

# Read /etc/passwd via traversal
curl "http://localhost:8080/view?filename=../../../etc/passwd"
```

### Secure Code

Uses `java.nio.file.Path` with `.normalize().toAbsolutePath()` to resolve all `..` sequences **before** the prefix check, making the validation reliable.

```java
@GetMapping("/view")
public ResponseEntity<Resource> readFile(@RequestParam String filename) {
    Path path = Paths.get(BASE_PATH).resolve(filename).normalize().toAbsolutePath();
    if (!path.startsWith(Paths.get(BASE_PATH).toAbsolutePath())) {
        throw new SecurityException("Access Denied!");
    }
    if (!path.toFile().exists()) {
        return ResponseEntity.status(404).body(null);
    }
    return ResponseEntity.ok()
            .contentType(MediaType.TEXT_PLAIN)
            .body(new FileSystemResource(path.toFile()));
}
```

## Dockerfile: Root vs Non-Root

Running as root means a path traversal bug can read **any** file on the system. Running as a dedicated user limits the blast radius.

=== "Insecure (runs as root)"

    ```dockerfile
    FROM eclipse-temurin:17-jdk-alpine

    WORKDIR /app

    COPY target/*.jar app.jar

    RUN echo "flag{P4th_Tr4v3r5s4l_Bug_F0und!}" > /flag.txt

    RUN mkdir -p /app/src/main/resources/static \
        && echo "This is a safe file." > /app/src/main/resources/static/public.txt

    EXPOSE 8080

    ENTRYPOINT ["java", "-jar", "app.jar"]
    ```

    !!! danger "Problem"
        No `USER` directive — the app runs as **root**. A path traversal exploit can read `/flag.txt`, `/etc/shadow`, and any other file.

=== "Secure (runs as appuser)"

    ```dockerfile
    FROM eclipse-temurin:17-jdk-alpine

    WORKDIR /app

    COPY target/*.jar app.jar

    RUN echo "flag{P4th_Tr4v3r5s4l_Bug_F0und!}" > /flag.txt

    RUN mkdir -p /app/src/main/resources/static \
        && echo "This is a safe file." > /app/src/main/resources/static/public.txt

    RUN addgroup -S appgroup && adduser -S appuser -G appgroup

    RUN chown -R appuser:appgroup /app

    USER appuser

    EXPOSE 8080

    ENTRYPOINT ["java", "-jar", "app.jar"]
    ```

    !!! success "Fix"
        A non-root user **cannot** read `/etc/shadow` even if the path traversal bug exists — defense in depth.

## Key Takeaways

- Never trust user input when constructing file paths
- Use `Path.normalize().toAbsolutePath()` to resolve traversal sequences **before** validation
- `java.io.File` path checks can be unreliable — prefer `java.nio.file.Path`
- Always run containers as a **non-root user** to limit the impact of file-read vulnerabilities
- Defense in depth: fix the code **and** harden the container
